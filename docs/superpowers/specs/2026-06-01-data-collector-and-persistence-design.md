# 設計: 日経225 データ収集デーモン ＋ ローカル永続化（基盤）

日付: 2026-06-01
ステータス: ドラフト（レビュー待ち）
対象サブプロジェクト: **SP1（基盤）** — 全体ロードマップの第1段

---

## 1. 背景と目標

現在 Finance_Monitor は **一切永続化していない**。価格(1分足/tick)・アラート履歴・相関はすべて
in-memory で、再起動で全消失する。in-memory の 1 分足は ~520 本(≈8.7時間)のみ。

このため:
- 再起動直後はウォームアップに最大65分かかり、上値/下値メドや z-score 検知が出ない
  （v0.3.36 で節目フォールバックを入れたが対症療法）。
- 日足・週足・複数日の節目/前日高安/ピボット等、**長スパンのメドが構造的に出せない**。
- アラートの良し悪しを実データで評価できず、閾値チューニングが体感頼み。

**最終目標**: テクニカルの予想精度を上げ、**上値/下値メドを“どのスパンでも”正しく出す**。
そのためには **多時間軸の連続した履歴** が必須 → ローカルに永続データ基盤を持つ。

ユーザーの構想（統合形）: 「日経225のデータ蓄積アプリを常駐させ、モニターアプリにデータを渡す」
＋「モニターを OFF にしたら重い API は止めるが、データ蓄積は続ける」。

---

## 2. 全体ロードマップ（分解）

| SP | 内容 | 本spec |
|----|------|--------|
| **SP1** | **収集デーモン ＋ ローカルSQLite ＋ 1分足/tick 永続化 ＋ モニターがDBから読む ＋ ON/OFF** | ★ |
| SP2 | 多時間軸メド生成モジュール（任意スパンの上値/下値メドを DB から）＋ UI/AI 統合 | 後続 |
| SP3 | アラート履歴 ＋ 事後値動き(+5/15/30分)記録 ＋ チューニング分析 | 後続 |
| SP4 | 予測系（時間帯シーズナリティ・ADRベースのメド・簡易バックテスト/ボラ連動閾値）| 後続 |

本 spec は **SP1（基盤）** のみを対象とする。SP2〜4 は基盤の上に乗る別 spec。

## 3. アーキテクチャ

```
┌────────────────────────┐        ┌──────────────────────┐
│  収集デーモン (collector) │ ──書込→ │  ローカル SQLite (WAL) │ ←─読込── ┌────────────────────┐
│  独立アプリ・常時稼働      │        │  jp225.db             │          │ モニター (Tauri,ON/OFF)│
│  feed 2秒ポーリング        │        │  ticks / bars_1m / meta│          │ DB から履歴+ライブ取得 │
│  → tick/1分足を追記        │        └──────────────────────┘          │ 検知/メド/AI/UI       │
│  起動時に履歴 backfill     │                                          └────────────────────┘
└────────────────────────┘
```

**役割分担**:
- **収集デーモン = 唯一の feed ポーラ兼 DB ライタ**。常時稼働。ユーザーの「収集アプリ常駐」に対応。
- **モニター = DB コンシューマ**。ライブ値も履歴も DB から読む。ON/OFF は「重い処理(AI/ニュース/
  検知/UI更新)」の制御であって、収集は止めない（収集はデーモン側で常時）。

この分担により「OFF で API を切るがデータは貯まる」が自然に成立する（モニターOFF≠収集停止）。

### 3.1 なぜ単一ポーラ（収集デーモンのみが feed を叩く）か
- nikkei225jp feed への礼儀（多重ポーリング回避）。
- 単一の真実源（モニターと収集で値がズレない）。
- ユーザーの構想（収集→モニターへ渡す）に一致。
- リスク: 収集が落ちるとモニターが盲目になる → モニターは「収集オフライン」を検出して
  ステータス表示（Yドット相当）。**フォールバックでモニター自身が feed を叩く機能は SP1 では持たない**
  （YAGNI、必要なら後続で追加）。

## 4. DB エンジンとスキーマ

### 4.1 エンジン: `node:sqlite`（Node ビルトイン）
- dev/ビルドは **Node v24**。SEA は同梱 node バイナリで動くため `node:sqlite` が **native モジュール
  無し**で使える（`better-sqlite3` の native ビルド/同梱問題を回避、最小依存方針に合致）。
- WAL モードで「単一ライタ(収集) + 複数リーダ(モニター)」を安全に同時実行。
- 注意点(実装時に確認): `node:sqlite` は experimental 扱い。起動フラグ(`--experimental-sqlite` 等)が
  要るか、SEA blob 経由で問題なく require できるかを**最初のタスクで PoC 検証**する。
  → 万一 SEA で使えない場合の代替: `better-sqlite3`(native) か、日付分割 JSONL ファイル + 自前集計。

### 4.2 スキーマ（SP1 範囲）
```sql
-- 生 tick（短期検知・直近のローリング窓用。保持は短め: 例 直近 3 日でローテーション/間引き）
CREATE TABLE ticks (
  symbol TEXT NOT NULL,
  t      INTEGER NOT NULL,   -- epoch ms (収集側の wall-clock)
  price  REAL    NOT NULL,
  PRIMARY KEY (symbol, t)
);

-- 1 分足（長期保持。メド/相関/検知 baseline の本体）
CREATE TABLE bars_1m (
  symbol TEXT NOT NULL,
  t      INTEGER NOT NULL,   -- 分頭 epoch ms
  o REAL, h REAL, l REAL, c REAL,
  PRIMARY KEY (symbol, t)
);

CREATE TABLE meta ( key TEXT PRIMARY KEY, value TEXT );  -- schema_version, last_backfill 等
```
- 対象 symbol は当面 **NIY=F(日経)** を主軸に、リアルタイム feed の全銘柄(NQ/YM/^HSI/CL/^TNX/JPY=X)も
  同じ表に格納（横断は将来使う）。
- 5分/1時間/日足/週足は **保存せず bars_1m から SQL 集計**（`GROUP BY` で任意スパン）。
- アラート/事後値動き/相関は **SP3** で表追加。

## 5. 収集デーモン（collector）

- **新規・独立アプリ**。バージョンは **0.4.00 から**（モニターの 0.3.36 系とは別系統の新ライン）。
- 実装: 既存 `server/sources/nikkei225jpFeed.ts`(feed 取得) を再利用した小さな Node プロセス。
  esbuild で単一 CJS → SEA で単一 exe 化（モニターと同じビルド方式）。
- ループ: 2 秒ごとに `fetchFeedPrices()` → 各 symbol を `ticks` に INSERT、分境界で `bars_1m` を upsert。
- **backfill**: 起動時に Yahoo chart API(`fetchMinuteBars`)で過去分(range=1d/利用可能分)を取得し
  `bars_1m` の欠損を埋める。さらに日足等の長期 backfill も将来 SP で拡張（SP1 は直近 1d で十分）。
- **常駐方式**: Windows のスタートアップ登録（自動起動）＋トレイ常駐。ユーザーが明示停止しない限り常時稼働。
- 単純・堅牢を最優先（feed→DB 以外の責務を持たせない）。

---

## 6. モニター側の統合

- 起動時に DB から `bars_1m`(直近 N 時間)と `ticks`(直近数分)を読み、**feedBars / tick バッファを
  即ウォームアップ**（65分待ち解消）。
- ライブ: priceLoop を「feed 直叩き」から「DB の最新 tick を読む」に変更（収集デーモンが ~2 秒ごとに
  書く前提）。検知(tickDetector/evaluateRealtime)・相関・上値下値メドは DB ベースに。
- **収集オフライン検出**: DB の最新 tick が一定時間(例 30 秒)更新されなければ「収集停止」をステータス表示。
- **ON/OFF トグル**: OFF で AI/ニュース/検知/UI更新など重い処理を止める（feed は元々モニターは叩かない）。
  ON/OFF 状態は configStore に保存。
- 既存の Yahoo フォールバック(`yahooChart`)は当面 backfill とデーモン障害時の保険として残置。

---

## 7. データフロー（通常時）

1. 収集デーモン: feed 取得 → `ticks` INSERT、分境界で `bars_1m` upsert（2秒周期、常時）。
2. モニター(ON): ~2秒ごとに DB 最新 tick を読み → SSE で UI 更新 / tickDetector / evaluateRealtime /
   メド生成（bars_1m を多時間軸集計）/ 必要時 AI。
3. モニター(OFF): DB 読みと重い処理を停止。収集デーモンは継続 → データは貯まり続ける。

---

## 8. 並行性・運用

- SQLite WAL: 収集=単一ライタ、モニター=リーダ。書込は短いトランザクションでまとめる。
- DB ファイル位置: ユーザーデータ領域（例 `%APPDATA%/jp225-monitor/jp225.db`）。両アプリで共有。
- 肥大対策: `ticks` は直近数日でローテーション/間引き、`bars_1m` は長期保持（1銘柄 ~525,600行/年 と軽量）。
- バックアップ/破損対策: WAL チェックポイント、起動時 integrity_check（簡易）。

---

## 9. エラーハンドリング

- feed 取得失敗 → 収集は前回値を書かずスキップ（穴は backfill / 次回で補完）、ログ。
- DB ロック競合 → WAL でほぼ回避。`busy_timeout` 設定。
- モニター: DB 読込失敗/収集オフライン → ステータス表示し、最後に読めた値を stale 表示。

---

## 10. テスト

- collector: パーサ→DB 書込の純粋部分を `node:sqlite` の in-memory DB(`:memory:`)で単体テスト。
  分境界での bars_1m upsert、tick 重複(PK)耐性、backfill 欠損埋め。
- monitor: DB から feedBars/tick へのウォームアップ復元、収集オフライン検出のしきい値。
- 既存の検知/メドのテストは DB 注入(getBars 差し替え)で維持。

## 11. SP1 の完了条件（Definition of Done）

- 収集デーモン(0.4.00)が常駐し、`ticks`/`bars_1m` を SQLite に 24/7 蓄積できる。
- モニターが DB から起動時ウォームアップし、ライブ値も DB 経由で取得して従来どおり動く
  （65分待ち・再起動データ消失が解消）。
- ON/OFF で重い処理を止めても収集は継続。
- `node:sqlite` が SEA 同梱で動くことを PoC で確認済み（ダメなら代替を spec 更新）。
